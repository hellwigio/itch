# Post-mortem отчет по восстановлению MediaWiki

## 1. Проблема

На сервере была развернута MediaWiki. При открытии сайта веб-сервер отвечал, но приложение показывало ошибку конфигурации: была проблема с переменной `$wgServer`.

Это означает, что сам `httpd` изначально работал, но MediaWiki была настроена некорректно.

После попытки перезапустить `httpd` сервис перестал запускаться. Позже выяснилось, что причина — полностью заполненный диск.

## 2. Проверка веб-сервера

Подключились к серверу по SSH:

```bash
ssh -i ich.pem ec2-user@35.159.74.144
# -i — путь к приватному SSH-ключу
```

Проверили состояние веб-сервера:

```bash
sudo service httpd status
# sudo — запуск от администратора
# service — управление сервисами
# httpd — Apache web server
# status — показать состояние сервиса
```

На первой проверке сервис был запущен:

```text
active (running)
```

То есть веб-сервер работал. Это также подтверждалось тем, что браузер получал ответ от MediaWiki, а не ошибку соединения.

После этого попробовали перезапустить сервис:

```bash
sudo service httpd restart
# restart — перезапустить сервис
```

После `restart` сервис не смог подняться. Проверка статуса показала `failed`.

Попробовали отдельно остановить и запустить:

```bash
sudo service httpd stop
# stop — остановить сервис

sudo service httpd start
# start — запустить сервис
```

Но `start` тоже не сработал. Это означало, что появилась системная проблема, мешающая Apache стартовать.

## 3. Обнаружение нехватки места

При попытке использовать автодополнение через `Tab` появилась ошибка:

```text
cannot create temp file: No space left on device
```

Это указывает, что на диске закончилось свободное место.

Проверили диск:

```bash
df -h
# df — показать использование файловых систем
# -h — human readable, размеры в удобном виде
```

Результат показал, что основной раздел занят на 100%:

```text
/dev/nvme0n1p1   10G   10G   0   100%   /
```

Из-за этого Apache не мог запускаться и писать логи.

## 4. Поиск больших файлов

Для поиска больших файлов использовали `find`:

```bash
sudo find / -type f -size +120M -exec du -h {} +
# / — искать от корня файловой системы
# -type f — только файлы
# -size +120M — больше 120 MB
# -exec du -h {} + — показать размер найденных файлов
```

Был найден большой лог веб-сервера:

```bash
/var/log/httpd/access_log
```

Он занимал около 7 GB.

Также в директории `/var/log/httpd` лежали архивы `.tar.gz`, что позже помогло найти проблему с неправильным бэкапом.

Проверили содержимое директории:

```bash
sudo ls -alh /var/log/httpd
# -a — показать все файлы
# -l — подробный список
# -h — размеры в удобном виде
```

## 5. Освобождение места

Так как старый `access_log` занимал почти весь диск, его удалили:

```bash
sudo rm -f /var/log/httpd/access_log
# rm — удалить
# -f — force, без подтверждения
```

После этого снова проверили диск:

```bash
df -h
```

Свободное место появилось. Проблема с диском была временно устранена.

## 6. Восстановление LocalSettings.php

MediaWiki требовала корректный конфигурационный файл `LocalSettings.php`.

Резервная копия лежала в домашней директории пользователя:

```bash
/home/ec2-user/LocalSettings\ \(5\).php
# обратные слэши экранируют пробелы и скобки в имени файла
```

Файл нужно было скопировать в директорию MediaWiki и переименовать:

```bash
sudo cp /home/ec2-user/LocalSettings\ \(5\).php /var/www/html/MediaWiki/LocalSettings.php
# cp — копировать файл
```

Проверили, что файл появился:

```bash
sudo ls -alh /var/www/html/MediaWiki/
```

После этого снова запустили Apache:

```bash
sudo service httpd start
```

Сервис успешно запустился:

```bash
sudo service httpd status
```

Результат:

```text
active (running)
```

## 7. Исправление переменной `$wgServer`

После восстановления `LocalSettings.php` сайт начал открываться, но происходил редирект на старый IP-адрес:

```text
18.153.51.162
```

А актуальный IP сервера был:

```text
35.159.74.144
```

Открыли конфигурационный файл:

```bash
sudo nano /var/www/html/MediaWiki/LocalSettings.php
# nano — консольный текстовый редактор
```

Нашли строку:

```php
$wgServer = "https://18.153.51.162";
```

Исправили на актуальный адрес:

```php
$wgServer = "http://35.159.74.144";
```

Также `https` заменили на `http`, потому что для сервера не был настроен сертификат.

После изменения конфигурации перезапустили Apache:

```bash
sudo service httpd restart
```

После этого MediaWiki стала открываться корректно.

Причина ошибки с IP: EC2-инстанс использовал динамический публичный IP. После перезапуска сервера Amazon выдал новый IP, а в конфиге MediaWiki остался старый.

## 8. Поиск причины переполнения диска

После восстановления сайта нужно было найти, почему диск заполнился.

Проверили cron текущего пользователя:

```bash
crontab -l
# -l — показать задачи cron
```

У `ec2-user` задач не было.

Проверили cron пользователя root:

```bash
sudo crontab -l
```

Там была найдена неправильная задача:

```bash
* * * * * tar -czf /var/log/httpd/log_$(date +\%Y\%m\%d).tar.gz -C /var/log/httpd .
```

Она запускалась каждую минуту.

Проблема в том, что архив создавался внутри той же директории, которая архивировалась:

```bash
-C /var/log/httpd .
```

То есть архив `/var/log/httpd/log_...tar.gz` попадал внутрь следующего архива. Получался рекурсивный бэкап: архивы архивировали сами себя и быстро росли.

## 9. Отключение неправильного cron-задания

Открыли root crontab:

```bash
sudo crontab -e
# -e — редактировать cron
```

Неправильную строку удалили или закомментировали:

```bash
# * * * * * tar -czf /var/log/httpd/log_$(date +\%Y\%m\%d).tar.gz -C /var/log/httpd .
```

Проверили результат:

```bash
sudo crontab -l
```

После этого задача больше не запускалась каждую минуту.

## 10. Скрипт для корректного бэкапа логов

Создали отдельный скрипт:

```bash
sudo nano /home/ec2-user/backup_logs.sh
```

Содержимое скрипта:

```bash
#!/bin/bash
# выполнять скрипт через bash

LOG_DIR="/var/log/httpd"
# директория с логами Apache

LOG_FILE="access_log"
# файл access log

BACKUP_DIR="/home/ec2-user/backups"
# директория для архивов, отдельно от /var/log/httpd

MAX_DAYS=3
# хранить архивы 3 дня

DATE=$(date +%Y%m%d)
# текущая дата для имени архива

ARCHIVE="$BACKUP_DIR/$LOG_FILE-$DATE.tar.gz"
# полный путь к архиву

mkdir -p "$BACKUP_DIR"
# -p — создать директорию, если ее нет, без ошибки если уже существует

tar -czf "$ARCHIVE" "$LOG_DIR/$LOG_FILE"
# -c — создать архив
# -z — сжать gzip
# -f — указать имя файла архива

truncate -s 0 "$LOG_DIR/$LOG_FILE"
# -s 0 — очистить лог, установив размер файла в 0 байт

find "$BACKUP_DIR" -type f -name "$LOG_FILE-*.tar.gz" -mtime +"$MAX_DAYS" -exec rm {} \;
# найти архивы старше MAX_DAYS и удалить
```

Сделали скрипт исполняемым:

```bash
sudo chmod +x /home/ec2-user/backup_logs.sh
# +x — добавить право на выполнение
```

Проверили запуск вручную:

```bash
sudo /home/ec2-user/backup_logs.sh
```

Проверили, что архив появился:

```bash
ls -alh /home/ec2-user/backups
```

Проверили содержимое архива:

```bash
tar -tf /home/ec2-user/backups/access_log-YYYYMMDD.tar.gz
# -t — показать содержимое архива
# -f — указать файл архива
```

## 11. Добавление нового cron-задания

Добавили корректный запуск скрипта один раз в сутки:

```bash
sudo crontab -e
```

Новая cron-задача:

```bash
0 0 * * * /home/ec2-user/backup_logs.sh
# запуск каждый день в 00:00
```

Теперь лог архивируется не каждую минуту, а один раз в день, и архивы складываются в отдельную директорию.

## 12. Root cause

Основная причина инцидента — неправильная cron-задача для бэкапа логов.

Она запускалась каждую минуту и создавала архив внутри `/var/log/httpd`, то есть внутри той же директории, которую архивировала. Из-за этого архивы попадали в следующие архивы, их размер быстро рос, и диск заполнился на 100%.

Из-за заполненного диска Apache не мог писать в `access_log` и после перезапуска не смог стартовать.

Дополнительная проблема — в `LocalSettings.php` был прописан старый IP-адрес в переменной `$wgServer`. После перезапуска EC2-инстанса публичный IP изменился.

## 13. Что было исправлено

* Проверили, что `httpd` изначально работал.
* После неудачного `restart` нашли причину падения сервиса.
* Обнаружили, что диск заполнен на 100%.
* Нашли большой `/var/log/httpd/access_log`.
* Удалили старый лог и освободили место.
* Скопировали резервный `LocalSettings.php` в директорию MediaWiki.
* Исправили `$wgServer` на актуальный IP.
* Перезапустили Apache.
* Нашли и отключили неправильную cron-задачу.
* Написали новый скрипт для ежедневного бэкапа логов.
* Добавили корректный запуск скрипта в cron.

## 14. Рекомендации

1. Использовать `logrotate` вместо самодельного скрипта.

`logrotate` — стандартный инструмент Linux для ротации логов. Он умеет архивировать, сжимать и удалять старые логи.

2. Не хранить архивы в той же директории, которая архивируется.

Иначе можно получить рекурсивный бэкап.

3. Не запускать бэкап логов каждую минуту.

Для access log достаточно ежедневной ротации.

4. Настроить мониторинг свободного места.

Например, отправлять предупреждение при заполнении диска на 80–90%.

5. Использовать Elastic IP или доменное имя.

Тогда после перезапуска EC2-инстанса не придется вручную менять `$wgServer`.

## 15. Итог

MediaWiki была восстановлена и снова открывается корректно.

Главная причина инцидента — неправильно настроенный cron для бэкапа логов, который создавал рекурсивные архивы и заполнил диск. Дополнительно была исправлена конфигурация MediaWiki: переменная `$wgServer` указывала на старый IP-адрес.
